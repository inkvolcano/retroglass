# RetroGlass — menu redesign brief

Paste this into Claude Design (or any design tool) as the starting context. It describes what
the app is, what is wrong with the menus today, everything the new design must hold, and the
constraints the implementation cannot escape.

---

## The product

RetroGlass is an Android multi-console emulator front-end (37 systems: NES through Dreamcast,
PS2, PSP). Its distinguishing feature is **XREAL glasses / external-display mode**: the game
renders on the glasses at full size while the phone becomes either a touch gamepad or a stats
dashboard. It is a personal, enthusiast-grade app — dense with options by design, not a
simplified console UI.

## The problem

Every menu is a stock `AlertDialog`. There are **44 of them**:

| | Emulation screen | Library screen |
|---|---|---|
| dialogs | 26 | 18 |

Breakdown of how they present: `setItems` flat list ×20, `setSingleChoiceItems` ×12,
`setMultiChoiceItems` ×2, custom `setView` ×9, plain message ×7.

The worst offender is the in-game main menu: **a single flat scrolling list of ~22 rows**, no
grouping, no icons, no current-value display, no hierarchy. To change one filter setting you
tap through Menu → Video filter → a category → an item, each step a separate grey system
dialog that fully covers the game. Related settings (Video filter / Combine filters / Upscale
factor / Filter settings / Saved looks — five separate entries) are scattered as siblings
among save-state, cheats, controller and exit actions.

Concretely, today's in-game menu is this flat list:

```
Save state · Load state · Fast forward: Off · Screen size (display) · Video filter ·
Combine filters… · Upscale factor… · Filter settings… · Saved looks… ·
Internal resolution ×2 (toggle) · Core options (system settings) · Cheats · Screenshot ·
Swap disc · Choose controller layout · Edit layout (move / resize) · Turbo / autofire… ·
Phone panel… · Rumble (vibration): Off · Controllers / players ·
Controller type (analog / lightgun…) · Display & extras · Reset game · Save & exit
```

## What to design

Two surfaces, consistent with each other.

**A. In-game menu** (overlays a running game; the game keeps rendering behind it)

Natural groupings — the designer should feel free to disagree, this is one reading:

- *Play* — save state, load state, fast forward, reset, screenshot, swap disc, save & exit
- *Video* — video filter, combine filters, upscale factor, filter settings, saved looks,
  internal resolution ×2, screen size, bezel, display & extras
- *Controls* — controller layout, edit layout, turbo/autofire, rumble, controllers/players,
  controller type, phone panel mode
- *System* — core options (a long, core-generated key/value list), cheats

**B. Library menus** — backup/restore, add source, scan results, top menu, screen-mode picker,
per-surface config, player picker, stick tuning, button remap, per-game options, BIOS status.

## Control types the design must cover

These all exist today and cannot be dropped:

1. **Action row** — fires immediately (Save state, Screenshot, Reset)
2. **Toggle row** — on/off, label currently mutates ("Rumble: Off")
3. **Single-select list** — 17 video filters, 4 upscale factors, layouts, bezels
4. **Multi-select list** — "Combine filters" (13 checkboxes, *order matters* — the chain is
   composed in a fixed pipeline order, and the UI should communicate that it is a chain)
5. **Slider** — filter parameters (sharpness, bloom, scanline strength), 0.0–1.0
6. **Free-text value entry** — core options are generated by the emulator core at runtime;
   values are typed strings (e.g. `gliden64|angrylion|parallel`). Unbounded count: N64 alone
   exposes ~60 options. This is the ugliest surface and most needs a designer.
7. **Nested navigation** — filters are already 3 levels deep (Menu → Video filter →
   category → filter)
8. **Status/info display** — BIOS present/missing per console, disc/track info
9. **Destructive confirm** — delete game, reset all options to defaults

Also needed: a **current-value display**. Today you cannot see which filter is active without
opening the picker. Rows should show their state inline.

## Existing visual language (match this)

The app already has one hand-built panel that looks right — `CompanionView`, the phone
dashboard shown in glasses mode. Use it as the anchor:

```
BG      #0B0B12   panel background
TILE    #171722   card / row surface
STROKE  #2A2A3A   1px card border
FG      #EDEDF5   primary text
DIM     #8A8AA6   secondary text / labels
ACCENT  #9BE870   live values, active state (also the FPS readout, #B6FF6A)
```

Broader app palette:

```
#141419  window background / status bar / nav bar
#0B0B0E  library background
#1B1B22  search field, pill controls
#1C1C1E  panel
#2A2A2E  row surface (most-used colour in the codebase)
#3A3A41  borders
#7C6CF2  Material colorPrimary (purple) — currently under-used
```

Shape and type: cards are `GradientDrawable` rounded rects, **12dp** radius (pills 14dp).
Typography is system default with `Typeface.DEFAULT_BOLD` for headings — **no custom font
file exists**. The library uses a pixel-art "RETRO GLASS" wordmark as a raster logo. Each
console has its own brand accent colour (e.g. `#C0392B`, `#2E86C1`, `#E67E22`) used on
carousel art and initials chips — an opportunity to tint the in-game menu per console.

## Hard constraints (please respect — these are not negotiable)

1. **Android Views, built in Kotlin code. No Jetpack Compose, no XML layouts.** The entire UI
   is constructed programmatically. Compose is not a dependency and adding it is out of scope.
   Available: `androidx.appcompat`, `material:1.12.0`, `recyclerview`, `constraintlayout`.
   So: deliver a design that is expressible as nested `LinearLayout`/`FrameLayout`/
   `RecyclerView` with `GradientDrawable` backgrounds. Avoid blur, complex gradients,
   custom-shaped clipping, and heavyweight motion.
2. **Must be fully gamepad-navigable.** In glasses mode there is no touchscreen in front of
   the user's eyes. Today every dialog gets a `gamepadNavigable()` helper mapping
   A → select, B → back. The new menus need explicit focus states, a clear focus ring, and
   D-pad traversal order. **Focus visibility is a first-class requirement, not a polish item.**
3. **The game keeps running behind the menu.** It must stay partly visible (players use menus
   to compare filter settings against the live picture). Favour a side panel or partial-height
   sheet over a full-screen takeover. Anything that dims the game should dim it gently.
4. **Three display contexts**, all must work:
   - phone portrait (touch pad occupies the bottom ~55% of the screen)
   - phone landscape
   - external display + phone as pad/dashboard
5. **Display cutout safe area.** Portrait has a centred punch-hole camera; overlays must
   offset by the cutout inset (just fixed for the in-game overlays).
6. **One-handed reach in portrait.** The phone is being held as a controller — menu targets
   should sit in the lower two-thirds, not the top.
7. **Touch targets ≥ 48dp**, and the current dialogs' ~144px row height is actually about
   right for thumb use.
8. minSdk 26, targetSdk 35, dark theme only, **arm64 phone form factor** (dev device is
   1080×2640).

## What I'd like back

1. **In-game menu**, portrait and landscape: the root, one drill-down level, and a settings
   panel with sliders. Show the focused/gamepad state explicitly.
2. **A pattern library** rather than 44 bespoke screens: row types (action, toggle,
   single-select, multi-select, slider, text-value, status), a header, a nav/back affordance,
   and how nesting is expressed (drill-down? column? accordion?).
3. **The core-options screen** — the hardest case: ~60 machine-generated rows, long
   pipe-delimited value lists, arbitrary strings. A generic pattern that degrades gracefully.
4. **The combine-filters screen** — 13 toggles whose *order* is meaningful; ideally reads as
   a pipeline/chain, not a checklist.
5. Redlines in **dp**, and colours as hex, so it maps directly onto programmatic views.

## Open questions worth a designer's opinion

- Should the in-game menu be a **left side panel** (gamepad-friendly, keeps the game visible,
  works in both orientations) or a **bottom sheet** (thumb-friendly in portrait, worse in
  landscape)?
- Is a **two-column layout** (categories left, items right) better than drill-down for
  gamepad use? It halves the button presses but is tight in portrait.
- Should frequently-changed video settings get a **quick-access overlay** (a small HUD toggled
  by a hotkey) separate from the full menu?
- How much should the **per-console accent colour** drive the menu's look?
