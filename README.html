RetroGlass

A retro console emulator for Android whose game plays on any external display —
glasses, monitor, or TV — while your phone becomes the controller.
What it is, in one line: connect any external display — USB-C/DisplayPort glasses (XREAL, Rokid…), a USB-C or HDMI monitor or TV, or a wireless/Cast display — and the game renders there on a big screen while the phone shows a touch gamepad; the same trick VLC uses to put video on an external display. Disconnect it and the game hops back onto the phone automatically. It uses Android's DisplayManager + Presentation, so glasses and monitors are handled identically — and in-game screen rotation/size/position let you fit the picture to a portrait monitor or vertical arcade games.

Home screen (library)

The library is a console carousel: swipe (or tap the ◂ ▸ arrows beside the centre console) to pick a system, and the games for that system list below. Between the console and its name sits a control row with, on the left, a screen-mode selector (“Where to play”) and, beside it, a centred set of player slots — one per port the system supports (up to four). Each slot shows that system's control layout; occupied slots are lit and named (Touchscreen, or a connected pad), empty ports greyed out.

The screen-mode selector chooses where and how the game is shown:

  • Auto — follow the connected display and device rotation (the default).
  • On phone — portrait — game across the top, touch pad below it.
  • On phone — landscape — game in a centred window with the touch pad framing it (separate, not overlaid).
  • External display — game on the glasses/monitor/TV, the phone is purely the pad.
  • Fullscreen — game fills the phone in landscape with no touch pad; a connected physical gamepad drives it (only offered when a pad is present).

Supported systems

Emulation runs on LibretroDroid with official libretro cores bundled inside the APK (arm64). Sort the list alphabetically, by maker, or by release date (newest first), and click any system to expand its details.

Home computers (MSX, C64, Amiga, ZX Spectrum, Amstrad) have a joystick + fire layout. Full typing isn't possible through the emulation layer (LibretroDroid has no keyboard-device path), but VICE/PUAE/fuse each include their own on-screen keyboard you navigate with the gamepad. Arcade romsets are imported as .zip intact and must match FBNeo's romset version; Neo Geo needs neogeo.zip as BIOS. Several systems need a BIOS (Saturn, 3DO, ColecoVision, Intellivision, Amiga Kickstart, Neo Geo) — the BIOS button shows what's present/missing.

Controller layouts — landscape (full pad, no screen)

The external-display layout: the game is on the glasses/monitor so the whole phone is the pad — D-pad and sticks on the left, the console’s buttons on the right (spread wider and larger than the with-screen frame), Start/Select in the centre. D-pad/stick slots, the symmetric face cluster, the workable minimum sizes and the zero-overlap guarantee all match the with-screen frame. 31 distinct layouts across 40 systems.

Controller layouts — portrait with screen

Phone held upright, playing on the phone: the game is a band across the top and the touch pad fills the region below (the same per-console pad as the portrait gallery). Design rule: the pad region is kept at H/W ≈ 1.35–1.42 — on squarer (18:9) phones the app shrinks the game band so the pad keeps that shape; letting the region get shorter than ≈1.29 squashes arcs/diagonals and buttons start to touch. Portrait without a screen (pad fills the whole upright phone) has even more room and is clean at every phone aspect. Representative set shown; all 31 pads are in the portrait gallery above.

Controller layouts — landscape with screen

The On phone — landscape layout: the game plays in the centred window (dashed) and every control is pushed into the side columns that frame it — nothing over the screen, no overlapping controls. Rendered from the app’s layout coordinates. 31 distinct layouts across 40 systems.

Controller layouts — large displays (foldable & tablet)

Near-square foldable main screens (~1.1) and 16:10 tablets (~1.6) held in landscape. The same landscape layouts apply — the only change is a button-size cap (sizing is capped at ≈0.46× the width) so buttons stay phone-sized instead of ballooning with the taller short-side. With that, both the with-screen frame and the full pad are overlap-free from 1.1 through 2.44. (Foldable portrait/unfolded reading mode would use the portrait split.)

Widescreen (16:9) — one layout, flexible screen

Consoles/games that run 16:9 (Dreamcast, PS2, PSP, and cores with a widescreen option/hack) don't need a different controller layout. Because every control sits in the side columns, the game window keeps its fixed width and just takes the aspect from its height — 16:9 renders shorter (more letterbox top/bottom), 4:3 taller — and the pad never moves. Fixing width (not height) is what keeps a wider picture from reaching the controls. Dashed slot = fixed width; solid box = the actual picture.

How the extended-display mode works

The defining feature. When the app detects an external display it moves the game there and turns the phone into a full-screen controller.

The mechanism

  • An android.app.Presentation window is shown on the external display. On a phone that mirrors its screen by default (all of them, over USB-C DisplayPort), a Presentation automatically replaces the mirrored image — so the glasses show only the game, and the phone stays fully touchable as the pad.
  • External displays are detected via MediaRouter's live-video route and DisplayManager, filtering out internal panels — important on foldables like the Galaxy Z Flip, whose cover screen otherwise looks like an external "presentation" display.
  • There is exactly one emulator instance (the native core is a process-wide singleton). On plug/unplug the single game view is reparented between the phone and the glasses — the emulator is never torn down, so gameplay continues seamlessly. This also survives Android handing out a fresh display id on every reconnect.
No XREAL SDK is needed — the glasses appear as a standard 1920×1080 external monitor. Audio auto-routes to the glasses as a USB audio device. The only real-world catch: while the glasses occupy the USB-C port the phone can't charge, so a USB-C hub with power passthrough is recommended for long sessions.
Wireless displays add lag. Wired displays (USB-C/DisplayPort/HDMI) are near-zero latency and are the intended experience. Miracast / "Smart View" wireless displays work — they register as a normal external display — but add ~100–300 ms of input lag, noticeable in fast games. Chromecast via Google Cast is not supported: Cast streams pre-encoded media, not a live game surface, so there is no display for the game to render onto.

Controller

Each system gets a controller styled to match its real pad, drawn natively (no images) so it scales cleanly: NES's horizontal B–A, the SNES colour diamond (Y·X·B·A), the Genesis A/B/C arc, the DualShock shape-diamond with twin sticks, the N64 trident, GBA's side-by-side B–A, and so on. Every default layout is positioned to stay fully on-screen with no overlapping buttons on tall phones, and a safety clamp keeps any control (including the scaled presets and your own edits) from being clipped at the screen edge. Button glyphs are rendered at a uniform size across every button.

N64 layouts. The default N64 pad matches the real controller — one centred analog stick, a D-pad, big A/B, and the four yellow C-buttons (which drive the right analog). There's also an optional "Z in D-pad" preset with the Z button seated in the centre of an enlarged D-pad, so one thumb can hold a direction and Z together — slide onto the centre to latch Z, off to drop it.

Customise any layout

Menu → Edit layout: drag any control to move it, pinch to resize it. Changes are saved per layout, per system. Reset restores the layout's defaults.

Screen size

Menu → Screen size: a slider that scales the game image on the glasses (and a separate, smaller default when playing on the phone with no glasses, so the picture sits as a comfortable centered window).

Portrait & landscape

Rotate the phone freely. In portrait with no glasses, the game shows as a screen across the top and the controller sits below it (classic handheld layout); the size slider sets how tall the screen is. In landscape, the game is centered with the controller as an overlay. With glasses connected, either orientation works and the phone is a full controller.

Screen size, rotation & position

Menu → Screen size gives size, a Rotate button (0/90/180/270° — for vertical “TATE” shmups, or comfort on the glasses) and horizontal/vertical position sliders to nudge the picture when it isn’t filling the whole display. Reset restores straight and centered.

Three ways to play

External display + touch padThe headline mode — game on glasses/monitor/TV, phone is the controller.
External display + Bluetooth pad(s)Game on the big screen, one or two BT controllers play, phone is free.
Phone as the displayNo external screen: BT pad(s) play, the game fills the phone like a handheld / couch-co-op TV. Touch pad hides; an auto-fading ≡ button opens the menu.
Controller-only navigation. Every menu — in-game options and the game library — works from a Bluetooth gamepad, so you never put the pad down. Open the in-game menu with L1 + R1 + Select (or the ≡ button / back gesture); D-pad moves, A confirms, B goes back. Choose Save & exit and the library is D-pad navigable with A to launch — so "finish, pick another game, tweak options" is a controller-only flow, with or without an external screen.

Quality-of-life

  • Physical gamepad support — when a pad plays, the game fills the phone like a handheld and the touch pad hides; a floating ≡ button (and the L1+R1+Select hotkey) still reach the menu. Input works even with the game on the glasses.
  • Controllers / players — assign phone / BT1 / BT2 to any player port, with per-controller button remapping and left-stick-as-D-pad.
  • Controller type — where the core offers it, switch a port's device (PS1 digital / analog / DualShock, lightguns, N64 pad variants), saved per system.
  • Gyro aiming — tilt the phone to drive the right analog stick (aim), with adjustable sensitivity — pairs well with an external display.
  • Stick tuning & hotkey — per-controller dead zone / sensitivity, and a configurable in-game menu hotkey.
  • FPS counter and a screen background / bezel (dark / gradient / custom image) behind a shrunk picture.
  • Cover art — set a cover per game, or drop images next to ROMs and a folder scan adopts them.
  • Save-data backup / restore — export SRAM, save-states and settings to a .zip and restore them (survives uninstall / new phone).
  • Local crash reports — after a crash, the next launch offers to share a log; nothing is sent automatically.
  • Core options — each system's real libretro settings, e.g. PS1 2× internal-resolution upscale, PSP/N64 render scale, region.
  • Cheats — per-game GameShark / Action Replay / PAR code list, toggle on/off.
  • Screenshots — saved to Pictures/RetroGlass.
  • Video filters — Off / CRT scanlines / LCD grid / Sharp / smooth-edge upscale, from the menu.
  • Save-state slots (4 manual) plus auto-save on exit & auto-resume on launch, and native SRAM battery saves.
  • BIOS status — the BIOS button shows which systems have their BIOS and which are missing (with expected filenames).
  • Library — group by system or by maker (Nintendo / Sega / Sony / …) with a tap, plus a search box and favourites / recently-played.
  • Fast-forward (2×) and rumble → phone vibration toggles.
  • Auto-pause on glasses disconnect — the game freezes as a small screen with tap-to-resume.
  • Loading indicator while disc-based games spin up; per-system layouts & drag/resize editor.

Getting started

  1. Install — sideload RetroGlass.apk (allow "install unknown apps"). arm64 phones only.
  2. Add ROMs — tap Add ROMs → Pick files to multi-select games, or Scan a folder to import a whole tree at once. Everything is auto-sorted to the right system.
  3. PlayStation BIOS — tap Import BIOS and add a PS1 BIOS (scph1001.bin etc.) for best compatibility. Without it PS1 games run on an HLE BIOS that works for many but not all titles. (A folder scan picks up recognised BIOS files for you.)
  4. Play — tap a game. Plug the glasses in any time — before or during play — and the game moves to them.

Unified library: scan a folder

Add ROMs → Scan a folder walks the folder recursively and builds one tidy library:
  • Every ROM is routed to its system's folder by extension (and content, for ambiguous .bin).
  • Per-system subfolders (PS2/, Dreamcast/, Saturn/, 3DO/, PSX/) are read as hints, so shared .chd/.iso disc images land on the right console instead of all defaulting to PS1.
  • Recognised BIOS files (e.g. scph5501.bin, dc_boot.bin, sega_101.bin) are moved into the system directory automatically.
  • Non-game files (box art, readmes, save files) are ignored.

Multi-disc games

RetroGlass supports multi-disc games and auto-detects them. On import, a set named Game (Disc 1).chd, Game (Disc 2).chd, … (also "Disk"/"CD") is stitched into an .m3u playlist, so it shows as a single library entry — the individual discs are hidden. Switch discs mid-game from the menu → Swap disc. You can also drop in your own .m3u; the discs it lists are hidden and play through it.

Disc systems share file types

PlayStation, PlayStation 2 and Dreamcast all use .chd/.cue/.iso, which can't be told apart by filename — so shared disc images import as PlayStation by default (unless a subfolder hint says otherwise). Long-press a game → Change system to move it to PS2 or Dreamcast. Dreamcast .gdi/.cdi files are recognised automatically.

PAL PlayStation games & LibCrypt

Many PAL PlayStation games (e.g. Final Fantasy VIII PAL, SLES-02080) use LibCrypt copy protection, which hides data in the disc's subchannel. A plain .chd doesn't contain it, so the game boots to a black screen. Two fixes:
  • Use an NTSC-U (US) release — those have no LibCrypt and just work.
  • Import the matching .sbi subchannel file next to the disc (RetroGlass now accepts .sbi via Add ROMs and stores it alongside the game so PCSX-ReARMed picks it up), or use a .chd rebuilt from a Redump dump that includes subchannel data.

Error reporting

If a game fails to load (bad dump, missing BIOS, LibCrypt without .sbi, unsupported core), the error is shown on the glasses as well as on the phone — so you're not left staring at a black external screen wondering what happened.

Not possible with this engine

These are LibretroDroid limitations, not to-do items (confirmed by decompiling the 0.14.0 AAR): rewind, online netplay, and RetroAchievements have no API surface, and a hardware-keyboard path for home computers doesn't exist (sendKeyEvent is gamepad-only). Adding them would mean swapping the emulation engine for a RetroArch frontend, which would give up the clean reparent-based extended-display mechanism that makes the glasses mode work. Local co-op (two BT pads) works; cross-phone play does not.

Technical notes

  • Package: com.nvanloo.retroglass · minSdk 26 · target 35 · arm64-v8a
  • Emulation: LibretroDroid 0.14.0 (GLSurfaceView + Oboe audio + save states) from JitPack.
  • Cores: libretro builds in jniLibs/arm64-v8a, dlopened by path at runtime. For Play, all 32 are rebuilt from source with NDK r28+ so their LOAD segments are 16 KB-aligned (0x4000) — verified by scripts/check_16k.py. Saturn needed -O1 to dodge an r28 optimizer hang.
  • Input: routed per device by descriptor (InputConfig): each pad → a player port, with per-pad button remap and left-stick-as-D-pad. Menu hotkey L1+R1+Select; dialogs are made gamepad-navigable via each Dialog's own setOnKeyListener (A→confirm, B→back).
  • Core options: read live from GLRetroView.getVariables(); applied at load via GLRetroViewData.variables and live via updateVariables(). The allowed-value list isn't exposed by the library, so CoreOptions.KNOWN_VALUES ships curated pickers for the high-impact keys.
  • Licensing: LibretroDroid and the cores are GPL-3.0, so this build is GPL. That's fine for the Google Play Store as long as the source stays public (Lemuroid, GPL+libretro, ships on Play) — the real gate is the 16 KB alignment above, which is now met. Also distributable via GitHub Releases / F-Droid. ROMs/BIOS must be dumped from hardware you own.
